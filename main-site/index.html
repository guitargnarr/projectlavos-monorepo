Development Methodology

Four core principles guide all work at Project Lavos. These are detailed in the manifesto and explored further in The Verification Gap article.

  1. Substance first — Working software over impressive pitches. All 69 portfolio sites are deployed and working, not mockups or screenshots.
  2. Built to last — Good systems outlast good intentions. Clean architecture, modular components, proper testing.
  3. Measure twice — Test the work, then ship it. Playwright E2E testing, CI/CD pipelines, build verification before every deployment.
  4. People over tech — Technology works for humans, not the reverse. Accessible, performant, user-focused design.

Additional principles: "Show, don't tell" (results over claims), "Simple scales" (clarity over complexity), "Test the seams" (systems fail where they connect), "Tools serve people" (technology serves humans).

Template Tier System

Each client site is built on one of four proven tiers. Highland Cleaners demonstrates all four tiers for the same business.

  • Tier 1: Essential — Single-page landing with smooth scroll navigation, contact form, Google Maps, service cards. Ideal for businesses needing a clean web presence.
  • Tier 2: Professional — Multi-page with React Router, 3-step booking wizard, photo gallery with lightbox, form validation. Ideal for service businesses with appointment scheduling.

Guitar Scale Reference Library

24 guitar scale reference books available as downloadable PDFs. Each book covers 12 keys, 5 CAGED positions, 8 exercise types, and 480 total exercises with mathematically verified tablature. All tablature is generated by computational fretboard logic using Python — not hand-transcribed or AI-generated — because LLMs cannot reliably count frets or compute musical intervals.

How Matthew Works With AI

AI collaboration is central to the workflow, but with clear boundaries. AI handles implementation speed — scaffolding components, writing boilerplate, generating test cases, producing variations across 69 sites. Matthew handles all judgment calls — architecture decisions, design quality, business context, and verification that things actually work. LLMs cannot do reliable arithmetic or music theory, so quantitative work (scale calculations, financial formulas, data validation) always uses Python, not prompts. He maintains 85 custom-trained Ollama models for domain-specific tasks and uses Claude for production integrations. This approach is documented in The Verification Gap.
